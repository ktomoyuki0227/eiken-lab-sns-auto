## 1. プロジェクト概要

| 項目 | 内容 |
|------|------|
| 案件名 | 英検ラボ SNS 自動投稿システム |
| クライアント | 長畑さん（Groovy House 経由） |
| 開発担当 | 久保友幸（1人で進める） |
| 対象事業 | Nippon English School（NES）> 英検ラボ |
| 目的 | SNS への毎日投稿を自動化する（V1：Instagram / note、最終的には全SNS対象） |
| 開発期限 | 2026年4月末（V1完成目標） |
| 開発プラットフォーム | Dify（ノーコード LLM ワークフロー） |

### Dify を選んだ理由

以下のツール比較をもとに、今回の要件に最も合致する Dify を採用した。

| ツール | 強み | 弱み | 今回の適合度 |
|--------|------|------|------------|
| Dify | ノーコード LLM ワークフロー専用。Knowledge（RAG）・スケジューラ・HTTP ノードが一体。LLM プロンプト管理が直感的。 | エンジニア以外には学習コストが少しある。 | ◎ 最適 |
| n8n | 汎用自動化ツール。豊富なインテグレーション。セルフホストで無料。 | LLM 専用機能が薄い。プロンプト管理・Knowledge が別途必要。 | △ LLM 用途には遠回り |
| Zapier / Make | ノーコード自動化。セットアップが簡単。 | 無料枠が非常に狭い。LLM ノードは別途連携が必要で複雑。有料プランは高額。 | △ コスト面で不利 |
| LangChain / LangFlow | LLM パイプラインに特化。柔軟性が高い。 | コード知識が必要。非エンジニアには難しい。運用保守コストが高い。 | △ 開発コストが高い |
| フルスクラッチ（Node.js 等） | 自由度が最大。ロジックを完全にコントロールできる。 | 開発・保守コストが最大。今回の納期・規模に不釣り合い。 | ✕ オーバースペック |

採用決定の根拠：
1. LLM プロンプト管理・Knowledge・スケジューラがノーコードで完結する
2. 長畑さん自身が Dify ワークスペースを確認・操作できる（透明性が高い）
3. 開発初期は友幸の Gemini 無料枠、最終検証は長畑さんの Claude クレジット3000円で低コスト運用が可能
4. 将来的に他コースへ横展開する際もワークフローを複製するだけで対応できる

## 2. 事業背景

Nippon English School（NES）は長畑さんが運営するオンライン英会話スクール。
傘下に4つの事業を展開している。

| 事業名 | 内容 |
|--------|------|
| 英検ラボ | 英検対策に特化（V1 の対象） |
| TOEIC GYM | TOEIC対策に特化 |
| 留学 | 留学準備向け英会話 |
| 英会話 | 一般英会話 |

今回のシステムは英検ラボ向けに構築するが、設計段階から「全プラットフォームへの転用」を前提とする。
V3 以降では英検ラボ以外の事業（TOEIC GYM / 留学 / 英会話）向けの SNS ネタ生成にも横展開する。
Dify のワークフローを複製して事業ごとのプロンプト・カテゴリに差し替えるだけで対応できるよう設計する。

## 3. 要件定義

### 3-1. 機能要件

| # | 機能 | 概要 |
|---|------|------|
| F1 | 情報収集 | 毎日定時に英検関連ニュース・時事ネタを自動収集 |
| F2 | コンテンツ生成 | LLM で Instagram 用台本5候補・note 用記事5候補を生成（計10候補/日） |
| F3 | ドラフト保存 | Google Drive のスプレッドシートに10候補を保存 |
| F4 | 人間によるレビュー | 担当者が1本を選んで投稿（完全自動化はしない） |
| F5 | 分析データ収集 | Instagram Graph API でインプレッション・リーチ等を取得（V2）※英検ラボ公式アカウントが必要 |
| F6 | ナレッジ蓄積 | 分析結果を Dify Knowledge に保存し、次回生成に反映（V2） |
| F7 | PDCAダッシュボード | Google Looker Studio で分析結果を可視化（V2） |
| F8 | 画像生成 | Instagram サムネイル画像を自動生成（V2） |

### 3-2. 非機能要件

- 開発初期は友幸の Gemini 学生無料枠を使用。最終検証フェーズは長畑さんが Dify に入れた Claude クレジット（3000円分）を使用（詳細は 6-3 参照）
- 長畑さんが Google スプレッドシートを開くだけで内容を確認できること
- 長畑さんがスプレッドシートを確認・編集できること
- 本番移行後は長畑さん側で Dify クレジットを管理できること

## 4. 投稿仕様

### 4-1. 対象SNSと展開フェーズ

最終的には以下の全SNSへの自動投稿を目指す。各SNSに最適化したフォーマットで生成する。

| SNS | フェーズ | 状況 |
|-----|---------|------|
| Instagram | V1（優先） | メイン。リール台本形式で対応 |
| note | V1（優先） | メイン。ブログ記事形式で対応 |
| X（Twitter） | V3以降 | 短文・拡散特化のフォーマットで対応予定 |
| LINE | V3以降 | LINE公式アカウント向け配信文を想定 |
| ブログ | V3以降 | SEO特化の長文記事形式で対応予定 |

### 4-2. 制作量・運用フロー（V1）

| SNS | 生成本数 | 投稿本数 | 確認者 | 保存先 |
|-----|---------|---------|-------|-------|
| Instagram | 5候補/日 | 1本/日 | 長畑さん | Google Drive |
| note | 5候補/日 | 1本/日 | 長畑さん | Google Drive |

運用フロー：
1. 毎朝8:00 に Dify が自動起動
2. Instagram 用5候補 + note 用5候補を生成
3. Google Drive のスプレッドシートに保存
4. 長畑さんがスプレッドシートを確認し、各SNSで1本ずつ選んで投稿

生成時刻：毎朝8:00 JST（Dify スケジュールトリガー設定値・変更可能）

### 4-3. ターゲット

英検合格を目指す中高生。英検勉強中だが継続できていない・やり方がわからない・何から始めればいいか迷っている層を主なターゲットとする。

note については、子どもの英検合格を支援したい親御さん向けのコンテンツも一定割合で生成する（子どもへの英語教育への投資を検討している保護者層）。

### 4-4. CTA（Call to Action）

CTA（シーティーエー）とは「Call to Action（行動喚起）」の略で、コンテンツを見た人に「次にしてほしい行動」を促す一言のことです。動画や記事の最後に入れることで、「見て終わり」にならず、体験申し込みや問い合わせなどの具体的なアクションにつなげる役割を持ちます。

全コンテンツ共通で以下を使用する。

```
無料体験レッスンまたはお問い合わせはこちら
→ https://eikenlab.jp
```

### 4-5. コンテンツカテゴリ（ローテーション）

5候補を生成する際は異なるカテゴリを混ぜ、多様性を持たせる。
媒体によって各カテゴリの使用割合を変化させる（例：Instagram は行動喚起系・悩み系を多め、note は試験対策系・親御さん系を多め）。

| カテゴリ | 訴求内容 |
|---------|---------|
| 悩み系 | 現状に共感→自分ごと化 |
| 習慣化系 | 継続できない原因の言語化→納得感 |
| 効率系 | 非効率な勉強法への不安→正しい方法への関心 |
| 試験対策系 | 英検特有のコツ・対策情報 |
| 行動喚起系 | 迷いを断ち切る→即行動 |
| 時事系 | 英検関連ニュース・試験日程と連動したネタ |
| モチベーション系 | 合格後の未来・理想像を想起させる |
| 親御さん系 | 子どもへの英語教育投資を後押しする内容（note 向け） |

#### カテゴリを増やしたい場合

操作手順：Dify のワークフロー内にある LLM ノードのプロンプトを編集し、カテゴリ一覧の箇所に1行追加するだけで対応できる。

運用フロー（長畑さんからリクエストがある場合）：
- リクエストはすぐに対応できないため、定期レビュー（月1回程度を想定）の形で意見を集める
- 把握でき次第、友幸が Dify のプロンプトを更新する
- 緊急性が高い場合は個別に相談

### 4-6. Instagram コンテンツフォーマット

※ 以下のフォーマットはまだ実績データがないため仮定義。実際に投稿を開始して反応を見ながら V2 以降で随時アップデートする。

プラットフォーム特性：縦型ショート動画（リール）。視覚・聴覚で瞬時に刺さる短いセリフが命。中高生に刺さる話し言葉・テンポ感を意識する。

出力する項目：

```
【タイトル】
動画のフックとなる一言（画面テキストに重ねる想定）

【台本】
各セクションを時間軸で区切る。セリフは短く・話し言葉で。

0〜3秒（Hook）：
  共感・衝撃・問いかけの一言
  例）「英検、また落ちた…」

4〜11秒（Agitate）：
  問題を深掘り。不安・共感をさらに喚起
  例）「やる気はあるのに、やり方がわからない」
  例）「なんとなく単語だけやっても、本番で点が取れない」

12〜18秒（Solution）：
  解決の方向性をチラ見せ。具体的すぎず、続きを見たくなるように
  例）「合格する人には、共通する勉強の順番がある」

最後2秒（CTA）：
  「無料体験レッスンまたはお問い合わせはこちら」

【キャプション（投稿本文）】
1〜2行のフックテキスト（本文冒頭に表示される部分）
空行
ハッシュタグ（10〜15個）
  英検関連：#英検 #英検対策 #英検勉強法 など
  ターゲット関連：#中学生 #高校生 #英語勉強 など
  拡散狙い：#英語学習 #受験英語 など
```

総尺の目安：15〜25秒

### 4-7. note コンテンツフォーマット

プラットフォーム特性：テキストブログ。検索経由で読まれることが多く、SEOと読みやすさが重要。中高生が「参考になった」と感じて最後まで読める情報量と構成を意識する。Instagram と違い、読者に実際の価値（知識・気づき）を提供することが目的。

ターゲット構成：中高生向けと親御さん向けを混在させる。目安は週3〜4本が中高生向け、週1〜2本が親御さん向け（子どもへの英語教育投資・英検の有効性を伝える内容）。

出力する項目：

```
【タイトル】
検索意識・悩み解決型のタイトル
  例）「英検3級に3回落ちた私が、4回目で合格できた理由」
  例）「英検のリスニングが聞き取れない人へ【これだけ変えた】」
  例）「英検2級を目指す高校生が、最初にやるべきたった1つのこと」

【リード文（100〜150字）】
読者の悩みに共感し、記事の価値を伝える導入文。
  例）「英検の勉強を始めたのに、全然点が伸びない…と悩んでいませんか？
  　　実は多くの人が、最初に間違った順番で勉強しています。
  　　今回は、合格者が共通してやっている勉強の流れを紹介します。」

【本文（800〜1200字）】
セクション見出しを使って読みやすく構成する。
  - 問題の深掘り（なぜ伸びないのか）
  - 具体的なアドバイス or 知識（英検に特化した内容）
  - 実践イメージ（中高生でも今日からできること）

【まとめ（3〜5行）】
記事の要点を箇条書きで整理。読み飛ばした人にも伝わるように。

【CTA】
「英検合格をサポートする無料体験レッスンを実施中です。
まずはお気軽にご相談ください。
→ https://eikenlab.jp」
```

## 5. システム構成

### 5-1. 全体アーキテクチャ

```
【日次：コンテンツ生成ワークフロー（毎朝8:00 JST）】※ V1から実装

Dify スケジュールトリガー
    ↓
Web Search ノード（Dify 標準）：「英検 ニュース」等のキーワードで最新情報を検索
  ※ V1 は Dify 標準の Web Search を使用（外部 API キー不要）
  ※ 英検公式X（旧Twitter）の投稿が情報鮮度の面で最も有効（確認済み）
  ※ 参照先URLは長畑さんから後日共有予定
    ↓
Knowledge 参照ノード：過去分析から「伸びるコンテンツの傾向」を取得（V2から有効）
    ↓
LLM ノード（Claude Haiku）：
  - Instagram用 5候補を生成（台本 + キャプション + ハッシュタグ）
  - note用 5候補を生成（タイトル + リード + 本文 + まとめ + CTA）
  ※ Claude を選定した理由：長畑さんが Dify に Claude クレジット（3000円分）を入れ済み。
    テキスト生成に特化した軽量モデル（Haiku）で十分な品質が出る見込み。
    開発初期は友幸の Gemini 無料枠で試作し、最終検証フェーズで Claude に切り替える。
    ↓
HTTP ノード：Google Drive のスプレッドシートに全10候補を保存
    ↓
長畑さんがスプレッドシートを確認 → Instagram 1本・note 1本を選んで投稿

─────────────────────────────────────────────

【週次：分析・ナレッジ更新ワークフロー（週1回）】※ V2から実装

Dify スケジュールトリガー
    ↓
HTTP ノード：Instagram Graph API から分析データを取得
    （インプレッション・リーチ・継続視聴数など）
    ↓
LLM ノード（Claude Haiku）：「今週の傾向まとめ」を文章化
    ↓
Dify Knowledge に保存
    （次回の日次ワークフローで自動参照される）

─────────────────────────────────────────────

【常時：PDCAダッシュボード（V2以降）】

Google スプレッドシート（分析データ蓄積）
    ↓
Google Looker Studio で可視化（無料）
```

### 5-2. データフロー

```
Web Search（Dify標準）→ Dify ワークフロー → Google スプレッドシート（ドラフト）
                              ↑                         ↓
                        Dify Knowledge            担当者がレビュー・投稿
                              ↑
                    Instagram Graph API（週次・V2）
```

## 7. Google / Instagram 設定方針

### 7-1. Google スプレッドシートへの書き込み

個人のGmailアドレスを直接共有する方法は使わない。サービスアカウント方式を採用する。

手順（概要）：
1. Google Cloud Console でプロジェクトを作成
2. Google Sheets API を有効化
3. サービスアカウントを作成（ロボット用Googleアカウント）
4. サービスアカウントのJSONキーをダウンロード
5. 長畑さんのスプレッドシートをサービスアカウントのメールアドレスと共有
6. Dify の HTTP ノードからサービスアカウントで認証して書き込む

長畑さん側の作業：スプレッドシートを1つ用意して、こちらから送るメールアドレスと共有するだけ。

### 7-2. Instagram アカウントの推奨（V2 で必要）

V1 では Instagram アカウントは不要（Google Drive 保存のみ）。
V2 以降で Graph API を使う際に以下を参考にアカウントを作成する。

Creator アカウント（クリエイターアカウント）を推奨する。

| 比較軸 | Businessアカウント | Creatorアカウント（推奨） |
|--------|-------------------|------------------------|
| 向いている用途 | 企業・店舗 | 個人・教育者・コンテンツクリエイター |
| Instagram Graph API | 利用可能 | 利用可能 |
| 分析ツール | 法人寄り | 個人向けで見やすい |
| 設定の手軽さ | やや複雑 | シンプル |

Graph APIを使うにはFacebookページとの連携が必要（無料・設定5分程度）。

### 7-3. Instagram Graph API トークン運用（V2）

Instagram Graph API を使用するには、英検ラボ公式の Instagram アカウントが必要。
アクセストークンの有効期限があるため、以下の運用が必要。

| トークン種別 | 有効期限 | 対応方法 |
|------------|---------|---------|
| 短期トークン | 約1時間 | 初回のみ取得。長期トークンへ即交換 |
| 長期トークン | 60日 | 60日ごとに更新が必要 |

運用フロー（V2 実装時）：
1. 英検ラボの Instagram アカウントを長畑さんに作成いただく（Creator アカウント推奨）
2. Facebook ページと連携（5分程度の初期設定）
3. 友幸がトークンを初回取得・Dify に登録
4. トークン期限切れの45日前にリマインド通知を設定（Google カレンダーなどで管理）
5. 期限前に長期トークンを再取得・更新（将来的には自動更新スクリプトで対応予定）

|---|--------|------|------|

| 5 | Dify DSL ファイル（バックアップ） | .yml ファイル | ワークフローのエクスポートファイル（復元・複製用） |

### 納品フロー

```
V1 開発完了
    ↓
① Google スプレッドシートを長畑さんに共有

    ↓

    ↓
③ 操作マニュアルを共有

    ↓
本番運用開始
```

※ 長畑さんの主な操作インターフェースは Google スプレッドシート。Dify への直接アクセスは不要。

### 運用開始後のサポート体制
